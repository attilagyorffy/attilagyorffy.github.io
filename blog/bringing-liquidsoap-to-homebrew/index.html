Bringing Liquidsoap to Homebrew: A Formula for Community Radio

A twenty-year-old streaming language with no Homebrew formula. That seemed worth fixing.

Developer Tools Infrastructure
Mar 22

If you have ever tried to run an internet radio station, you have probably hit the same wall. You start with a playlist — maybe a folder of MP3s and a script that feeds them to Icecast. It works, until it does not. The stream dies at 3 AM and nobody notices. Two tracks play back-to-back with wildly different volumes. There is no crossfade, no jingles, no way to cut to a live input when a DJ shows up. You bolt on more scripts, more cron jobs, more glue. It gets ugly fast.

This is the problem Liquidsoap solves.

What Liquidsoap is

Liquidsoap is a programming language designed specifically for audio and video streaming. Not a playlist manager. Not a config file for a streaming server. An actual statically typed language where sources of audio are first-class values that you can compose, transform, and route.

The idea started at the École Normale Supérieure de Lyon around 2004, when a group of students wanted a better way to run their campus radio. Twenty years later, it powers Radio France, AzuraCast, Radionomy, and countless community stations. The language has grown to handle video, HLS output, SRT ingest, speech synthesis, YouTube streaming, and more — but the core philosophy has not changed: simple things should be simple, complex things should be possible.

A basic Liquidsoap script reads like a description of what you want your radio to do: 

music = playlist("~/Music")
jingles = playlist("~/Jingles")
s = rotate(weights=[1, 4], [jingles, music])
s = crossfade(s)
output.icecast(%mp3, host="localhost", port=8000, password="hackme", mount="/stream", s)

That is a radio station. Jingles every four tracks, crossfaded transitions, streaming to Icecast in MP3. If any of these sources fail — say the music directory gets unmounted — Liquidsoap detects it at type-check time, before the script even runs. If you want a live input that takes priority over the playlist, you add a fallback. If you want time-based scheduling, there is switch. The type system ensures you cannot accidentally create a stream that might go silent.

This is fundamentally different from chaining together ffmpeg commands, cron jobs, and shell scripts. Liquidsoap understands audio at a semantic level — clocks, synchronisation, fallibility, track boundaries — and handles the hard parts so you do not have to.

Why it was not on Homebrew

Yet until now, there was no Homebrew formula for it. I decided to change that.

Liquidsoap is written in OCaml. That alone makes packaging non-trivial. It uses OPAM (OCaml's package manager) to pull in dozens of OCaml libraries at build time, dune as its build system, and has a complex relationship with system libraries like ffmpeg and libcurl. The build process is not a simple ./configure && make — it is closer to bootstrapping a small OCaml ecosystem inside a temporary directory, compiling everything, then extracting the binary and its runtime files.

There are only a handful of OCaml-based formulas in homebrew-core (semgrep, flow, zero-install), and each one has had to solve these packaging challenges independently. Liquidsoap adds its own wrinkles: the binary needs to find its standard library (.liq scripts) and unicode data (camomile) at runtime, and the default build mode bakes in paths that assume a Linux FHS layout (/usr/share/liquidsoap/).

Not exactly the kind of thing brew create handles for you.

Why I wanted a formula

I have been building custom audio pipelines and wanted to iterate on .liq scripts locally before deploying them to a server. Test the playlist logic, tweak the fallback chains, verify the encoding settings — the kind of thing you want a fast feedback loop for.

But installing Liquidsoap on macOS meant either wrestling with OPAM directly (slow, fragile, leaves a .opam directory in your home) or running the Docker image (works, but adds overhead and makes soundcard access painful on macOS).

What I wanted was simple: brew install liquidsoap, write a script, run it. The same workflow you would have with ffmpeg or any other command-line tool.

And beyond my own use case — every friction point in installation is a potential community radio builder who gives up before writing their first script. The easier Liquidsoap is to install, the more people experiment with it.

How the formula works

The formula follows the same OPAM-based pattern that semgrep and zero-install use, with some liquidsoap-specific adaptations.

Build strategy: patched POSIX mode. Liquidsoap has three build modes for resolving runtime paths. The "posix" mode hardcodes paths like /usr/share/liquidsoap/libs — obviously wrong for Homebrew. The formula patches these paths at build time to point to Homebrew's prefix (share/liquidsoap/libs, var/log/liquidsoap, etc.), then builds with LIQUIDSOAP_BUILD_TARGET=posix.

System OCaml, temporary OPAM root. Instead of building the OCaml compiler from source (which takes 10+ minutes), the formula uses Homebrew's OCaml package directly via --compiler=ocaml-system. OPAM gets a temporary root inside the build directory — nothing touches your home directory or persists after install.

The build steps boil down to:

  1. Patch runtime paths and a vendored library's dune file (the cry module references bytes, a compat library removed in OCaml 5.x)
  2. Initialize OPAM with the system compiler
  3. Install OCaml dependencies from the project's opam files
  4. Install the ffmpeg OCaml bindings
  5. Build with dune
  6. Install, relocate man pages and stdlib files to Homebrew-conventional locations
  7. Copy camomile unicode data from the OPAM switch before it is cleaned up

The whole thing builds in about 2.5 minutes on an M1 Max.

What you can do with it

After brew install liquidsoap, you have a working liquidsoap binary with ffmpeg support. Some things you can do right away:

Generate audio files: 

liquidsoap 'output.file(%wav, fallible=true, "/tmp/sine.wav", sine(duration=3.))'
afplay /tmp/sine.wav  # 440 Hz for 3 seconds

Play a directory of music files and encode to MP3: 

liquidsoap 'output.file(%ffmpeg(format="mp3", %audio(codec="libmp3lame")), fallible=true, "/tmp/radio.mp3", playlist("~/Music/MyAlbum"))'

Note: playlist does not recurse into subdirectories by default — point it at a directory that contains actual audio files, not a parent with nested folders.

Stream to an Icecast server: 

s = playlist("~/Music")
s = crossfade(s)
output.icecast(%ffmpeg(format="mp3"),
  host="localhost", port=8000,
  password="hackme", mount="/stream",
  s)

Set up an HLS output for web-based listening: 

s = playlist("~/Music")
output.file.hls("/var/www/radio",
  [("live.aac", %ffmpeg(format="adts", %audio(codec="aac")))],
  s)

Pull in a live HTTP stream with a local fallback: 

live = input.http("https://icecast.radiofrance.fr/fip-hifi.aac")
emergency = playlist("~/Music")
s = fallback(track_sensitive=false, [live, emergency])
output.file(%ffmpeg(format="mp3"), "/tmp/output.mp3", s)

The --build-config flag shows you exactly what is enabled. Currently the formula ships with ffmpeg for encoding/decoding, curl for HTTP, and the full liquidsoap standard library (2086 functions).

Room to grow

The current formula is intentionally minimal — ffmpeg covers the most important use case (encoding and decoding in virtually any format, including all the codecs you would typically install separate bindings for like opus, vorbis, or flac). But there are features that ffmpeg does not cover, and that I would like to explore adding in future updates:

  • LADSPA / Lilv — host external audio effect plugins (reverb, compressor, EQ from plugin libraries like LADSPA and LV2). FFmpeg has its own built-in filters, but these bindings let you use the wider ecosystem of audio plugins that studios and radio stations already rely on.
  • OSC — Open Sound Control protocol, for controlling the stream in real-time from external hardware or apps. Think adjusting volumes, triggering transitions, or switching sources from a phone or a MIDI controller.
  • Prometheus — export metrics from a running liquidsoap instance for monitoring. Essential for production deployments where you want to track buffer health, encoding performance, or source availability in Grafana.
  • sqlite3 — powers liquidsoap's built-in medialib.sqlite media library. It indexes your audio files' metadata (title, artist, album) into a persistent database and lets you build dynamic playlists from queries — "play a random jazz track that has not aired in the last hour." Without it, the media library works in-memory only and loses everything on restart.

Note that the formula does not output audio to your speakers directly — output() falls back to output.dummy without a native audio backend. For local testing, write to a file and play it back: 

liquidsoap 'output.file(%wav, fallible=true, "/tmp/test.wav", sine(duration=3.))'
afplay /tmp/test.wav

These optional bindings cannot be installed on demand after the fact. They get compiled into the liquidsoap binary at build time — there is no runtime plugin system. And homebrew-core does not support --with-foo build options (they were removed years ago to keep the build matrix manageable).

So if you need additional bindings today, there are a couple of paths:

  • Install via OPAM directly — full control over which bindings to include, at the cost of managing an OPAM switch yourself.
  • A third-party tap could offer a formula with build options (e.g. --with-ladspa, --with-prometheus) for users who want a customised build without leaving Homebrew entirely.

For most use cases, the Homebrew formula with ffmpeg is a solid baseline — and the features above are on my list to explore adding as default dependencies in future updates.

The takeaway

Community and amateur internet radio has a long history, and Liquidsoap has been at the heart of it for twenty years. But accessibility matters. Every friction point in installation is a potential hobbyist who gives up before writing their first .liq script.

Having brew install liquidsoap just work — with the standard library loading correctly, ffmpeg support enabled, and a clean 2.4.2 version — removes one of those friction points. The PR is up on homebrew-core, and once merged, anyone on macOS will be three words away from building their own radio.

If you have been curious about internet radio, there has never been a better time to start. The Liquidsoap book is an excellent resource, and the community Discord is welcoming. Write a script, point it at your music library, and see what happens.

  • brew install liquidsoap gives you a working binary with ffmpeg support — encode, decode, and stream in virtually any format
  • The formula builds from source in about 2.5 minutes using your system OCaml — nothing persists in your home directory
  • LADSPA, OSC, Prometheus, and sqlite3 support are candidates for future updates
  • The PR is up on homebrew-core — once merged, anyone on macOS is three words from their own radio
